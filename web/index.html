PROJECT RATIONALE

为什么这样设计

01 · 项目目标

从“景点清单”升级为受约束的决策系统

用户真正需要的不是更多推荐，而是在预算、天数、兴趣和体力之间做取舍，并知道方案为什么可行。

02 · 模型选择

GPT-5 mini 负责理解，工具负责算账

模型只用于偏好理解和说明文字；预算、路线、去重和强度由确定性 Python 工具计算，避免数字漂移。

03 · 数据集

5 城结构化 POI 种子库

每个景点含标签、片区、时长、票价、坐标和室内外属性；酒店与交通为规划参考区间，并明确提示复核。

04 · 用户思考

把不确定性展示出来

系统不会假装拥有实时房态。它区分“规划估算”和“可预订价格”，并给出天气来源、预算警告与替换优先级。
