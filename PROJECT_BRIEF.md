# Waypoint 项目阐述稿

下面是一套约 5–7 分钟的面试讲解顺序。

## 1. 项目背景

我观察到多数旅行 AI 的输出像一篇攻略：景点很多、语言流畅，但经常忽略总预算、同行人数、跨片区时间和真实预订的不确定性。用户真正想要的是一份可以执行和调整的决策方案。

因此我做了 Waypoint，一个只负责旅行规划的垂直 Agent。

## 2. 项目目标

用户输入预算、日期、天数、同行人数、兴趣、旅行节奏、住宿档位和交通偏好后，系统需要完成四件事：

1. 推荐与兴趣匹配的景点。
2. 将景点按片区和时长排成逐日路线。
3. 在总预算内给出酒店与交通档位。
4. 把超预算、单日过载和无法验证的要求明确提示出来。

我的成功标准不是“文案像真人”，而是结果可计算、可解释、可复现。

## 3. 用户与场景

核心用户是没有时间做大量攻略、但又不愿意完全跟团的年轻自由行用户。典型场景是：两个人从上海出发，三天预算 3600 元，喜欢历史、美食和摄影，希望节奏平衡。

用户的主要焦虑有三类：

- 不知道预算够不够。
- 不知道景点之间是否顺路。
- 不知道 AI 推荐的信息是否仍然有效。

产品分别用预算拆分、片区排程和数据边界提示解决。

## 4. Agent 设计

我没有让一个大模型直接输出整份行程，而是拆成七个步骤：

1. Preference Interpreter：标准化输入。
2. Destination Search：在 POI 知识库中按兴趣和约束检索。
3. Budget Allocator：计算交通、酒店、餐饮、门票和机动预算。
4. Route Optimizer：按片区和每天景点上限分组。
5. Weather Adapter：获取预报或季节性提示。
6. Constraint Validator：检查重复、超预算、强度和未验证要求。
7. Plan Composer：组装最终方案。

这样每一步都能单独测试，也能在模型不可用时回退。

## 5. 技术选型

前端使用原生 HTML、CSS 和 JavaScript，后端使用 Python 标准库。选择零依赖方案，是为了让面试官下载后能直接运行，也降低现场演示风险。

模型层选择可选的 `gpt-5-mini + Responses API`。模型只负责把已经确定的结构化方案解释得更自然，不修改景点、价格、日期和交通。这样既利用模型的语言能力，又避免金额和路线被模型随意改变。

如果没有 API Key，系统使用确定性模板生成说明，核心规划功能不受影响。

## 6. 数据集

当前使用自己整理的中国城市冷启动知识库，包括 5 个城市、40 个 POI、15 个酒店档位参考和 8 条主要城际连接。

POI 不只是名字，还包含兴趣标签、片区、时长、票价、坐标、室内外属性和基础优先级。路线优化和预算工具直接使用这些字段。

这不是训练数据，而是推理时知识库。实时天气使用 Open-Meteo，失败时回退到季节性提示。酒店和门票价格明确标为参考区间。

## 7. 产品取舍

第一，我没有伪装成预订平台。没有实时房态时，只给档位和参考范围。

第二，我把总预算定义为同行人总额，并展示六类费用，让用户能主动取舍。

第三，补充要求不一定能被机器验证。例如用户写“不吃辣”，系统会记录并提示人工确认，而不是假装所有餐厅都满足。

第四，兴趣、节奏和无障碍偏好真正进入排序公式，不只是放在 Prompt 里。

## 8. 验证

我为核心行为写了 8 个测试，包括：不能重复景点、茶文化兴趣应命中相关 POI、低预算必须预警、松弛节奏不能塞太多景点、低碳偏好应选择高铁，以及补充要求不能误报为已满足。

前端还需要在桌面和手机尺寸验证无横向溢出、文字遮挡和按钮错位。

## 9. 局限与下一步

目前路线距离使用片区近似，不是实时地图矩阵；酒店和交通不是实时库存。下一步会接入地图距离矩阵和可授权的预订数据，同时建立真实用户评估集，衡量预算误差、兴趣命中率和路线折返距离。

## 30 秒总结

Waypoint 的核心价值不是“AI 会写攻略”，而是把旅行规划拆成可验证的工具链：模型理解和解释偏好，数据集提供可追溯知识，确定性工具负责预算和路线，校验器负责诚实暴露风险。
